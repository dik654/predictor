# PulseAI 시스템 로직 — AI 요약용

## 슬라이드 1: PulseAI 개요

PulseAI는 편의점 POS 단말기의 이상을 실시간으로 탐지하고 미래 장애를 예측하는 시스템입니다.

핵심 기능 3가지:
- 실시간 이상 탐지: 지금 POS에 문제가 있는가?
- 미래 장애 예측: 1시간~48시간 뒤에 문제가 생길 것인가?
- 주변장치 모니터링: 결제 관련 장비가 정상 작동하는가?

---

## 슬라이드 2: 전체 아키텍처

데이터 흐름을 왼쪽에서 오른쪽으로 그려주세요:

1단계: POS 단말기 (C# Agent) → WebRTC로 5초마다 데이터 전송
2단계: Python Hub Server → 이상 탐지 엔진 2개 (ECOD + ARIMA) 실행
3단계: InfluxDB → 시계열 데이터 저장
4단계: React Dashboard → 실시간 시각화

## 슬라이드 3: 수집 데이터

POS에서 두 종류의 데이터를 수집합니다.

메트릭 데이터 (5초 간격, 상시 전송):
- CPU 사용률, 메모리 사용률, 디스크 I/O
- 네트워크 송수신량
- POS 프로세스 상태 (RUNNING/STOPPED)
- 소프트웨어 버전 정보

로그 데이터 (POS 유휴 시에만 비정기 전송):
- 주변장치 7개의 연결 상태 (동글, 핸드스캐너, 2D스캐너, 여권리더기, 충전기, 키보드, MSR)
- 결제 승인 처리시간 및 결과 (추후 추가)
- 영수증 프린터/용지 상태 (추후 추가)

## 슬라이드 5: 엔진 1 — ECOD (실시간 이상 탐지)

ECOD(Empirical Cumulative Distribution)는 "평소 패턴과 얼마나 다른가?"를 측정합니다.

작동 원리:
1. 최근 데이터로 "정상 패턴"을 학습합니다
2. 새 데이터가 들어오면 정상 패턴에서 벗어난 정도를 0~1 점수로 계산합니다
3. 점수가 높을수록 평소와 다릅니다

14개 지표를 동시에 분석합니다 (다변량 분석):

| 번호 | feature | 타입 | 값 범위 | 설명 |
|------|---------|------|---------|------|
| 1 | CPU | 연속 | 0~100% | CPU 사용률 |
| 2 | Memory | 연속 | 0~100% | 메모리 사용률 |
| 3 | DiskIO | 연속 | 0~ | 디스크 입출력 |
| 4 | NetworkSent | 연속 | 0~ bytes | 네트워크 송신량 |
| 5 | NetworkRecv | 연속 | 0~ bytes | 네트워크 수신량 |
| 6 | Process | 이산 | 0/1 | POS 프로세스 (0=STOPPED, 1=RUNNING) |
| 7 | Dongle | 이산 | -1/0/1 | 동글 (결제 핵심) |
| 8 | HandScanner | 이산 | -1/0/1 | 핸드스캐너 (상품 스캔) |
| 9 | PassportReader | 이산 | -1/0/1 | 여권리더기 |
| 10 | 2DScanner | 이산 | -1/0/1 | 2D스캐너 (QR/쿠폰) |
| 11 | PhoneCharger | 이산 | -1/0/1 | 휴대폰충전기 |
| 12 | Keyboard | 이산 | -1/0/1 | 키보드 |
| 13 | MSR | 이산 | -1/0/1 | 카드리더기 |
| 14 | POS_Idle | 이산 | 0/1 | POS 유휴 상태 (0=바쁨, 1=유휴) |

이산값 의미: 1=연결, 0=실패, -1=미사용 또는 미전송

핵심 가치 — 조합의 이상을 감지합니다:
- CPU 80% 단독 → 정상 (가끔 높을 수 있음)
- CPU 80% + 동글 실패 + 네트워크 급증 → 이상 (이런 조합은 본 적 없음)

심각도 판정:
- 0.9 이상 → Critical (즉시 조치 필요)
- 0.7 이상 → Warning (주의 관찰)
- 0.7 미만 → Normal (정상)

주기: 10초마다 실행

## 슬라이드 6: 이산값의 한계와 처리

ECOD는 "평소와 다른가?"만 판단하고, "좋은가 나쁜가?"는 모릅니다.

문제 예시:
- 학습 기간에 POS 프로세스가 대부분 꺼져(STOPPED) 있었다면
- 프로세스가 켜진(RUNNING) 것을 "평소와 다름"으로 판단합니다
- 하지만 켜진 것이 오히려 정상입니다

해결 방법:
- 이산값(주변장치, 프로세스)은 개별 이상 점수를 표시하지 않습니다
- 다변량 종합 점수에서만 사용합니다 (다른 지표와의 조합 이상만 감지)
- 대시보드에 개별 점수를 표시하는 항목은 연속값 5개뿐입니다: CPU, Memory, DiskIO, NetworkSent, NetworkRecv

## 슬라이드 7: 엔진 2 — AutoARIMA (시계열 예측)

AutoARIMA는 시계열 데이터의 추세와 패턴을 학습하여 미래 값을 예측합니다.

두 가지 역할:
1. 현재 시점 이상 탐지: 예측값과 실제값의 차이(잔차)가 크면 이상
2. 미래 예측: 1시간, 6시간, 12시간, 24시간, 48시간 뒤의 값을 예측

대상: CPU, Memory, DiskIO, NetworkSent, NetworkRecv (연속값만)

주변장치는 왜 ARIMA에 안 넣는가?
- 주변장치 값은 "연결(1) / 실패(0) / 미사용(-1)"뿐입니다
- "연결 → 연결 → 연결" 패턴에서 미래를 예측하는 것은 무의미합니다
- 시계열 추세 분석은 연속적으로 변화하는 값에만 의미가 있습니다

주기: 60초마다 실행

---

## 슬라이드 8: 미래 장애 예측 — 앙상블

ARIMA의 미래 예측값을 ECOD로 평가하여 "미래에 문제가 생길 것인가?"를 판단합니다.

흐름도를 그려주세요:

입력 1: ARIMA 미래 예측값 (CPU, Memory, DiskIO, NetworkSent, NetworkRecv → 연속값 5개)
입력 2: 현재 프로세스 상태 (1개)
입력 3: 현재 주변장치 상태 (7개)
입력 4: POS 유휴 상태 (1개)
이 네 입력을 합쳐서 14차원 벡터를 만듭니다

이 벡터를 장기 ECOD 모델(7일치 데이터로 학습)에 넣어서 ecod_score를 구합니다

별도로 주변장치 규칙 기반 점수(rule_score)를 계산합니다

최종 점수 = (0.8 x ecod_score + 0.2 x rule_score) x 예측 신뢰도

주변장치 상태 가정:
- 주변장치는 이산값이므로 미래 예측이 불가능합니다
- 현재 상태가 미래에도 유지된다고 가정합니다 (보수적 접근)
- 예: 지금 동글이 고장이면 48시간 뒤에도 고장으로 계산
- 장비가 수리되면 다음 데이터 수신 시 자동으로 상태가 갱신됩니다

---

## 슬라이드 9: 규칙 기반 점수 (rule_score)

ECOD가 데이터 부족으로 놓칠 수 있는 주변장치 장애를 보완하는 안전장치입니다.

ECOD와 규칙 기반의 관계:
- ECOD: 데이터에서 자동으로 이상을 학습 (80% 가중치)
- rule_score: 사람이 정한 도메인 지식 규칙 (20% 가중치)
- 둘이 중복으로 감지해도 문제없음 → 오히려 확신도가 올라가는 효과

장치별 실패 시 가산 점수:
| 장치 | 점수 | 이유 |
|------|------|------|
| 동글 | +0.5 | 결제 불가 |
| 핸드스캐너 | +0.4 | 상품 스캔 불가 |
| 2D스캐너 | +0.3 | QR/쿠폰 불가 |
| 기타 4개 | 각 +0.1 | 부분 기능 장애 |
| 2개 이상 동시 실패 | 최소 0.7 | 복합 장애 |

미사용(-1) 장치는 점수 0 (정상 취급)

---

## 슬라이드 10: 최종 점수 계산과 심각도

공식:
최종 점수 = (0.8 x ECOD점수 + 0.2 x 규칙점수) x 예측 신뢰도

예측 신뢰도란?
- ARIMA 예측이 과거에 얼마나 정확했는지에 따라 0.1~1.0으로 자동 조정
- 1시간 예측: 기본 0.8 (비교적 정확)
- 6시간: 0.6
- 12시간: 0.5
- 24시간: 0.4
- 48시간: 0.3 (불확실)
- 실제 오차 데이터가 쌓이면 자동으로 보정

심각도 판정:
- 최종 점수 0.9 이상 → Critical (즉시 조치)
- 최종 점수 0.7 이상 → Warning (주의 관찰)
- 그 외 → Normal (정상)

## 슬라이드 13: 예시 시나리오

시나리오 1 — 실시간 이상 탐지:
- 14:00 CPU 85%, 동글 실패, 네트워크 급증
- ECOD 다변량 점수 0.92 (Critical)
- 대시보드: "복합 지표 이상 심각" 경고 표시
- 조치: 매장 확인 필요

시나리오 2 — 미래 장애 예측:
- ARIMA가 6시간 뒤 CPU 90%로 예측
- 현재 동글 실패(0) 상태 유지 가정
- 장기 ECOD 평가: "이 조합은 7일간 본 적 없음" → 점수 0.85
- rule_score: 동글 실패 +0.5
- 최종: (0.8 x 0.85 + 0.2 x 0.5) x 0.6 = 0.47 → Normal
- 신뢰도(0.6)가 점수를 낮춰서 과잉 경보 방지

시나리오 3 — 주변장치 모니터링:
- 2D스캐너 14:00에 "실패" 수신
- 대시보드: "미연결 — 2시간 전부터" 표시
- ECOD 다변량: 다른 지표와 조합해서 종합 이상도 평가

## 슬라이드 14: POS 유휴 상태 판단

POS가 유휴 상태인지를 자동으로 판단하여 ECOD feature에 포함합니다.

판단 기준:
- Process=RUNNING + 주변장치 데이터 수신됨 → 유휴 상태 (POS가 주변장치 체크를 수행할 여유가 있음)
- Process=RUNNING + 주변장치 데이터 미수신 → 비유휴 상태 (POS가 결제 처리 등으로 바쁨)

활용:
- ECOD가 "유휴인데 CPU가 비정상으로 높음" 같은 이상 조합을 감지할 수 있습니다
- "비유휴(바쁜 상태)에서 CPU 높음"은 정상이지만, "유휴에서 CPU 높음"은 이상일 수 있습니다

## 슬라이드 15: Feature Engineering — 시스템 정교화의 핵심

현재 시스템은 14차원 ECOD로 동작합니다. 하지만 이것으로 완성이 아닙니다. 실제 운영 데이터를 관찰하면서 feature를 추가하고 조정해야 시스템이 점점 정교해집니다. 이것이 도메인 지식 기반 Feature Engineering입니다.

현재 14개 feature:
CPU, Memory, DiskIO, NetworkSent, NetworkRecv, Process, 동글, 핸드스캐너, 여권리더기, 2D스캐너, 충전기, 키보드, MSR, POS유휴상태

향후 추가 가능한 feature:
| feature | 기대 효과 |
|---------|----------|
| 결제 승인 처리시간 | "처리시간 3초 + CPU 90%" 조합의 이상 감지 |
| 요일/시간대 | 평일 점심 vs 심야의 정상 패턴 차이 반영 |
| 공휴일 flag | 명절/연휴의 특수 트래픽 패턴 구분 |
| 계절 | 여름/겨울 전력 부하 차이 |
| 영수증 프린터/용지 상태 | 출력 장애 사전 감지 |

feature가 추가될수록 ECOD가 더 정교한 "정상 범위"를 학습하게 됩니다.

## 슬라이드 16: 장기 ECOD 모델 — 학습과 재학습

장기 ECOD 모델은 "이 POS의 1주일간 정상 패턴"을 학습합니다.

학습 데이터: 최근 7일치 데이터 (InfluxDB에서 조회)
해상도: 5분 단위로 리샘플링 (약 2,016개 데이터 포인트)
최소 데이터: 100개 포인트 (약 8시간분)
재학습 주기: 24시간마다, 매일 새벽 3:30에 실행 (매장 운영에 영향 없는 시간대)
데이터 보존: 30일 (이후 자동 삭제)

7일 윈도우의 의미:
- 월~일 전체를 포함하여 주중/주말 패턴 차이를 학습
- 새벽~심야까지 시간대별 패턴도 자연스럽게 포함

현재 한계:
- 공휴일이 학습 기간에 없으면 공휴일 패턴을 모름 → 정상인데 이상으로 판단할 수 있음
- 계절 변화(여름→겨울)는 7일 윈도우로는 반영 안 됨
- 이런 한계는 요일/공휴일/계절 feature를 추가하면 해결 가능

## 슬라이드 16-1: 데이터 부족 시 처리 (콜드 스타트)

신규 POS를 설치한 직후에는 7일치 학습 데이터가 없습니다. 이 경우 3단계 fallback으로 처리합니다.

1단계 — 설치 직후 ~ 약 8시간:
- 학습할 데이터가 부족합니다
- "CPU 90% 이상 = 위험" 같은 단순 규칙으로만 판단합니다
- 예: 오전 10시에 새 POS 설치 → 오후 6시까지는 단순 규칙 모드
- 한계: "CPU 75% + 동글 실패" 같은 조합 이상은 잡지 못합니다

2단계 — 약 8시간 ~ 7일:
- 쌓인 데이터로 ECOD를 학습합니다. 조합 이상 탐지가 시작됩니다
- 하지만 데이터가 적어서 정확하지 않을 수 있습니다
- 예: 설치 3일째, 평일 데이터만 학습됨 → 토요일의 낮은 CPU를 "이상"으로 오판할 수 있음

3단계 — 7일 이후 (정상 운영):
- 월~일 전체 패턴을 학습 완료
- 주중/주말, 시간대별 차이를 알고 있어 가장 정확한 판단이 가능합니다
- 예: "평일 점심 CPU 70%는 정상이지만, CPU 70% + 동글 실패 + 네트워크 급증 조합은 본 적 없다 → 이상"

핵심: 단계 전환은 자동입니다. 데이터가 쌓이면 다음 재학습 시점(매일 새벽 3:30)에 자동으로 더 정확한 단계로 올라갑니다. 사람이 개입할 필요가 없습니다.

## 슬라이드 17: 추후 확장 계획

추가 예정 데이터:
| 데이터 | 활용 계획 |
|--------|----------|
| 결제 승인 처리시간 | ECOD feature 추가, 결제 지연 패턴 감지 |
| 결제 승인 처리결과 | 승인 실패율 추적 |
| 영수증프린터 커버상태 | 주변장치 상태에 포함 |
| 영수증 용지상태 | 주변장치 상태에 포함 |

확장 용이성:
- 새 데이터 추가 시 수정할 파일 4개만 (매핑 추가 + 버퍼 추가 + ECOD 차원 확장 + 차트 추가)
