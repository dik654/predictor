# PulseAI — 시스템 로직 설명

## 개요

PulseAI는 편의점 POS 단말기의 **실시간 이상 탐지** 및 **미래 장애 예측** 시스템입니다.

```
POS 단말기 (C# Agent)
    │
    │ WebRTC DataChannel (5초 간격)
    ▼
Python Hub Server
    ├─ 실시간 이상 탐지 (ECOD + ARIMA)
    ├─ 미래 장애 예측 (ARIMA → ECOD 앙상블)
    └─ InfluxDB 저장
         │
         ▼
    React Dashboard (실시간 시각화)
```

## 2. 실시간 이상 탐지

메트릭 데이터가 도착할 때마다 두 개의 엔진이 동작합니다.

### 2-1. ECOD (다변량 이상 탐지)

**주기**: 10초마다

**원리**: ECOD(Empirical Cumulative Distribution)는 다변량 데이터의 분포를 학습하고, 새 데이터가 학습된 분포에서 벗어난 정도를 점수화합니다.

**입력 (14차원 벡터)**:
```
[CPU, Memory, DiskIO, NetworkSent, NetworkRecv, Process,
 동글, 핸드스캐너, 여권리더기, 2D스캐너, 충전기, 키보드, MSR,
 POS유휴상태]
```

| 번호 | feature | 타입 | 설명 |
|------|---------|------|------|
| 1~3 | CPU, Memory, DiskIO | 연속 | 시스템 리소스 |
| 4~5 | NetworkSent, NetworkRecv | 연속 | 네트워크 트래픽 |
| 6 | Process | 이산(0/1) | POS 프로세스 실행 여부 |
| 7~13 | 주변장치 7개 | 이산(-1/0/1) | 연결/실패/미사용 |
| 14 | POS_Idle | 이산(0/1) | POS 유휴 여부 (주변장치 데이터 수신 = 유휴) |

**출력**:
- 종합(Multivariate) 이상 점수 (0~1)
- 개별 메트릭별 이상 점수 (13개)
- 심각도: normal / warning / critical

**동작 방식**:
1. 최근 N개 데이터로 ECOD 모델 학습 (슬라이딩 윈도우)
2. 최신 데이터 포인트의 이상 점수 계산
3. 점수 정규화 (학습 데이터 대비 백분위)
4. 심각도 판정:
   - score > 0.9 → **critical** (상위 10%)
   - score > 0.7 → **warning** (상위 30%)
   - 그 외 → **normal**

**핵심 가치**: 단일 메트릭이 아닌 **조합**의 이상을 감지합니다.
예) CPU 80% 단독은 정상이지만, CPU 80% + 동글 실패 + Network 급증 조합은 이상.

**이산값(주변장치/프로세스)의 한계와 처리**:
ECOD는 "평소와 다른가?"만 판단하며, "좋은가 나쁜가?"는 모릅니다. 학습 데이터에서 Process가 대부분 STOPPED(0)였으면, RUNNING(1)이 "평소와 다름"으로 높은 점수를 받지만 실제로는 정상입니다. 주변장치 연결(1)도 마찬가지입니다.

따라서 **이산값은 per-metric 개별 점수 표시에서 제외**하고, **다변량(Multivariate) 종합 점수에만 사용**합니다. 조합의 이상(예: 동글 실패 + CPU 급등)은 의미 있지만, 개별 이산값의 "상위 몇%"는 의미가 없기 때문입니다. 대시보드에서 개별 이상 점수를 표시하는 항목은 CPU, Memory, DiskIO, NetworkSent, NetworkRecv (연속값 5개)뿐입니다.

### 2-2. AutoARIMA (시계열 예측)

**주기**: 60초마다

**원리**: AutoARIMA는 시계열 데이터의 추세/계절성을 학습하여 다음 값을 예측합니다. 예측값과 실제값의 차이(잔차)가 크면 이상입니다.

**대상 메트릭**: CPU, Memory, DiskIO, NetworkSent, NetworkRecv (연속값만)

**출력**:
- 예측값 (arima_predicted)
- 잔차 = |실제값 - 예측값| (arima_deviation)
- 미래 예측: 1시간 / 6시간 / 12시간 / 24시간 / 48시간 후 값
- 심각도: normal / warning / critical

**동작 방식**:
1. 최근 데이터로 ARIMA 모델 학습 (해상도별 리샘플링)
2. 현재 시점 예측 → 실제값과 비교 → 잔차 계산
3. 적응형 임계값: 잔차 이력의 표준편차 × K
4. 미래 예측: 다중 해상도로 장단기 예측
   - 5분 해상도 → 1시간, 6시간 예측
   - 1시간 해상도 → 12시간, 24시간, 48시간 예측

**주변장치는 왜 ARIMA에 안 넣나?**
주변장치 값은 1/0/-1 이산값이라 시계열 추세 분석이 무의미합니다. (연결→연결→연결 패턴에서 미래 예측은 의미 없음)

## 3. 미래 장애 예측 (ARIMA + ECOD 앙상블)

### 구조

```
ARIMA 미래 예측값 (연속값 5개)
    + 현재 주변장치 상태 (이산값 7개)
    + 현재 프로세스 상태 (이산값 1개)
    = 13차원 벡터
        │
        ▼
장기 ECOD 모델 (7일치 데이터로 학습)
        │
        ▼
    ecod_score (0~1)
        │
        + 주변장치 규칙 기반 점수 (rule_score)
        │
        = combined_score
        │
        × 예측 신뢰도 (reliability)
        │
        = final_score → severity 판정
```

### 3-1. 장기 ECOD 모델

- **학습 데이터**: InfluxDB에서 최근 7일치 메트릭 + 주변장치 이력
- **재학습 주기**: 6시간마다
- **목적**: "이 에이전트의 정상 패턴"을 학습
- **평가**: ARIMA 예측값 + 현재 이산 상태를 넣어서 "미래 예측 상태가 정상 범위 밖인가?" 판단

**주변장치 상태 가정**: 주변장치(동글, 스캐너 등)는 이산값(1/0/-1)이므로 ARIMA로 미래 예측이 불가능합니다. 따라서 **현재 상태가 미래에도 유지된다고 가정**합니다. 예를 들어 지금 동글이 실패(0) 상태라면, 1시간 뒤~48시간 뒤 평가에서도 동글 실패로 계산합니다. 이 가정은 보수적(worst-case) 접근으로, 장비가 수리되면 다음 주변장치 데이터 수신 시 자동으로 상태가 갱신되어 이후 평가에 반영됩니다.

### 3-2. 규칙 기반 점수 (rule_score)

주변장치 실패에 대한 보조 점수. ECOD가 학습 데이터 부족으로 놓칠 수 있는 케이스를 보완합니다.

| 장치 | 실패 시 가산 | 이유 |
|------|-----------|------|
| 동글 | +0.5 | 결제 불가 |
| 핸드스캐너 | +0.4 | 상품 스캔 불가 |
| 2D스캐너 | +0.3 | QR/쿠폰 불가 |
| 기타 | +0.1 | 부분 기능 장애 |
| 2개 이상 동시 실패 | 최소 0.7 | 복합 장애 |

미사용(-1)은 점수 0 (정상 취급).

### 3-3. 최종 점수 합산

```
final_score = (0.8 × ecod_score + 0.2 × rule_score) × reliability
```

- **ecod_score** (80%): 데이터 기반 이상 판단
- **rule_score** (20%): 도메인 지식 기반 보완
- **reliability** (0.1~1.0): ARIMA 예측 정확도 이력에 따른 신뢰도 가중
  - 1시간 예측: 기본 0.8 (비교적 정확)
  - 48시간 예측: 기본 0.3 (불확실)
  - 실제 오차율이 쌓이면 자동 조정

### 3-4. 심각도 판정

| 조건 | 심각도 |
|------|--------|
| final_score ≥ 0.9 또는 (ECOD 이상치 + score ≥ 0.85) | **critical** |
| final_score ≥ 0.7 또는 (ECOD 이상치 + score ≥ 0.65) | **warning** |
| 그 외 | **normal** |

## 4. 데이터 저장 (InfluxDB)

모든 데이터는 InfluxDB에 시계열로 저장됩니다.

| Measurement | 내용 | 주요 필드 |
|------------|------|----------|
| metrics | 원시 시스템 메트릭 | cpu, memory, disk_io, network, process |
| anomaly_detection | 실시간 탐지 결과 | score, threshold, confidence, arima_predicted |
| arima_forecast | ARIMA 미래 예측값 | predicted_value (horizon별) |
| arima_ecod_ensemble_forecast_eval | 앙상블 예측 평가 | ecod_score, rule_score, final_score |
| peripheral_status | 주변장치 상태 | dongle, hand_scanner 등 (1/0/-1) |
| file_versions | 소프트웨어 버전 | 파일별 버전 문자열 |
| accuracy | 예측 정확도 검증 | actual_value, forecast_value, error_pct |

## 5. 대시보드 (React)

### 실시간 화면 구성

1. **ECOD 다변량 이상 점수**: 시스템/주변장치 그룹별 이상 점수 추이
2. **시스템 상태 분석**: 현재 이상 여부 + 개별 메트릭 바 그래프
3. **주변장치 연결 상태**: 7개 장치의 연결/미연결/미전송 표시
4. **AutoARIMA 예측 vs 실제**: 메트릭별 예측선 + 실제선 + 잔차 (영역 + 막대)
5. **탐지 히스토리**: 최근 탐지 결과 테이블

### 데이터 흐름

```
InfluxDB ←(2초 폴링)→ Python API ←(HTTP)→ React Dashboard
```

## 6. 추후 확장 계획

### 추가 예정 데이터

| 데이터 | BodyType | 활용 계획 |
|--------|----------|----------|
| 결제 승인 처리시간 | payment_approval_time | ECOD feature 추가 (결제 지연 감지) |
| 결제 승인 처리결과 | payment_approval_result | 승인 실패율 추적 |
| 영수증프린터 커버상태 | receipt_printer_cover | 주변장치 상태에 포함 |
| 영수증 용지상태 | receipt_paper_status | 주변장치 상태에 포함 |

### POS 유휴 상태 판단

POS가 유휴 상태인지 판단하는 로직이 ECOD feature로 포함되어 있습니다.

- **유휴(1)**: 주변장치 데이터가 수신됨 (POS가 유휴 상태에서 주변장치 체크를 수행)
- **비유휴(0)**: 주변장치 데이터 미수신 (POS가 결제 처리 등으로 바쁨)

이 feature가 있으면 ECOD가 "유휴인데 CPU가 높음" 같은 이상 조합을 감지할 수 있습니다.

### Feature Engineering과 정교화

현재 시스템은 14차원 ECOD로 동작하며, **실제 운영 데이터를 관찰하면서 feature를 추가/조정해야 정교해집니다**. 이것이 도메인 지식(Feature Engineering)의 핵심입니다.

예시:
- 결제 승인 처리시간이 추가되면 → "처리시간 3초 + CPU 90%" 조합의 이상을 감지
- 요일/시간대 feature 추가 → "평일 점심 vs 심야"의 정상 패턴 차이 반영
- 공휴일 flag 추가 → 공휴일 특유의 트래픽 패턴 구분

현재 고려하지 않지만 향후 추가 가능한 feature:
- **요일/시간대**: 평일 점심 vs 심야는 CPU/메모리 패턴이 다름
- **공휴일**: 명절/연휴는 평소와 다른 트래픽 → 별도 패턴 학습 필요
- **계절**: 여름(에어컨)과 겨울(난방)에 따른 전력/시스템 부하 차이

### 장기 ECOD 모델 학습 주기

- **학습 데이터**: InfluxDB에서 최근 **7일치** 데이터를 가져와 학습
- **재학습 주기**: 24시간마다, **매일 새벽 3:30** 에 실행 (매장 운영에 영향 없는 시간대)
- **데이터 보존**: InfluxDB retention 30일
- **해상도**: 5분 단위로 리샘플링 (7일 = 약 2,016 포인트)
- **최소 데이터**: 100개 포인트 (5분 해상도 × 100 = 약 8시간분)

7일 윈도우의 의미: 주중/주말 패턴을 모두 포함하여 "이 POS의 1주일간 정상 범위"를 학습합니다. 공휴일이나 특수 기간에는 학습 데이터에 해당 패턴이 포함되어야 정확한 판단이 가능합니다.

### 데이터 부족 시 처리 (콜드 스타트)

신규 POS 설치 직후처럼 7일치 데이터가 없는 경우, 시스템은 3단계로 점진적으로 정교해집니다.

**1단계: 설치 직후 ~ 약 8시간 (데이터 100개 미만)**

ECOD 학습이 불가능합니다. 이 기간에는 "CPU 90% 이상이면 Critical, 80% 이상이면 Warning" 같은 **고정 임계값**으로만 판단합니다. 단순하지만 최소한 심각한 상황은 잡을 수 있습니다. 다만 "CPU 80% + 동글 실패" 같은 조합 이상은 감지하지 못합니다.

> 예: 새 POS를 오전 10시에 설치. 오후 6시까지는 고정 임계값 모드로 동작.
> CPU가 92%까지 올라가면 Critical 경보. 하지만 CPU 75% + 동글 실패 조합은 잡지 못함.

**2단계: 약 8시간 ~ 7일 (데이터 100개 이상, 7일 미만)**

서버 메모리에 쌓인 데이터(또는 InfluxDB에 저장된 데이터)로 ECOD를 학습합니다. 조합 이상 탐지가 가능하지만, 학습 데이터가 적어서 "정상 범위"를 완전히 파악하지 못합니다. 예를 들어 평일 데이터만 있으면 주말 패턴을 모릅니다.

> 예: 설치 3일째. 평일 낮 시간대 데이터만 학습됨.
> 토요일에 CPU가 평소보다 낮아지면 "이상"으로 판단할 수 있음 (실제로는 주말이라 정상).

**3단계: 7일 이후 (정상 운영)**

월~일 전체 데이터로 ECOD를 학습합니다. 주중/주말, 시간대별 패턴을 모두 포함하여 가장 정확한 "정상 범위"를 알고 있습니다. 이 단계에서 14차원 조합 이상 탐지가 제대로 동작합니다.

> 예: 설치 10일째. 주중 점심 시간 CPU 70%는 정상으로 학습됨.
> 같은 시간에 CPU 70% + 동글 실패 + 네트워크 급증이 동시에 발생하면 "이 조합은 7일간 본 적 없다" → 이상 판정.

단계 전환은 자동입니다. 데이터가 쌓이면 다음 재학습 시점(새벽 3:30)에 자동으로 상위 단계로 올라갑니다.

### 확장 시 변경 사항

1. **influx_writer.py**: `DEVICE_NAME_MAP`, `BODY_TYPE_MAP`에 매핑 추가
2. **detector.py**: `MetricBuffer`에 deque 추가, ECOD matrix 차원 확장
3. **forecast_evaluator.py**: 학습/평가 벡터에 새 feature 추가
4. **Dashboard**: ECOD 차트에 새 메트릭 라인 추가

현재 구조는 새 데이터가 추가되면 위 4개 파일만 수정하면 됩니다.
